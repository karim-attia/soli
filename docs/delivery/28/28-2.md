## Requirements
- Reduce auto-queue delay values so auto-up runs much faster.
- Ensure auto-queue draw steps (stock→waste) remain snapshot-gated so the drawn card’s flight uses the correct origin.
- Keep the change scoped to auto-up; do not speed up manual interactions.
- Reduce perceived handoff delay between auto-up completion and celebration motion onset.
- Remove the initial “stacked kings” look by making celebration card motion visibly start immediately.
- Start the celebration only after the final winning card has visually arrived on its foundation pile so the last flight and yellow foundation glow do not overlap with the celebration.
- Remove the remaining end-of-auto-up jitter by keying the final win handoff off the actual winning-card settle event rather than only a fixed timeout.
- Keep the board visually stable during the pending win-handoff window instead of applying win cleanup as soon as `hasWon` flips.
- Make the empty tableau outlines fade out smoothly during the win cleanup instead of disappearing in one abrupt render commit.
- Defer non-visual win bookkeeping away from the exact `hasWon` edge so the final winning card has less JS-thread contention while its destination flight settles.
- Keep `yarn demo:auto-solve` aligned with the Android release signing path and the current demo logging contract.

## Implementation Plan
1. Add auto-up timing constants and wire them into `useAutoQueueRunner` from `useKlondikeGame`.
2. Ensure `dispatchWithFlight` snapshot-gates `ADVANCE_AUTO_QUEUE` draw steps by inferring the affected card id.
3. Validate the sequence via manual play and demo auto-solve.
4. Tune celebration launch ramp so motion starts earlier without changing overall celebration architecture.
5. Add a launch initial-progress floor so separation from the foundation stacks is visible on the first celebration frames.
6. Gate the celebration start behind the final foundation-arrival handoff so the winning move finishes visually before celebration motion begins.
7. Replace the timeout-only win handoff with an event-driven winning-card settle signal plus a guarded fallback.
8. Delay top-row win cleanup until after the pending handoff window so the board does not reconfigure mid-flight.
9. Repair `yarn demo:auto-solve` so it loads the release signing env and watches the current demo completion signals.
10. Investigate the remaining perceived lag at celebration start and determine whether the empty-slot cleanup is actually blocking or just visually abrupt.
11. Keep empty tableau outlines mounted through the handoff and fade them out once the visual win cleanup begins.
12. Defer solved-game bookkeeping and celebration-state assembly away from the exact `hasWon` frame while preserving the existing handoff fallback.
